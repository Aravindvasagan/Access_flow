Interview Answer:
"I create a Service Principal in Entra ID and store its credentials in Azure Key Vault. Using Unity Catalog, I create a Storage Credential backed by the Service Principal and an External Location pointing to ADLS. I grant Storage Blob Data Contributor to the Service Principal on ADLS. For Databricks access control, I use Unity Catalog permissions. HR receives SELECT only, while Developers receive SELECT and MODIFY. I do not grant OWNERSHIP, MANAGE, or ALL PRIVILEGES, preventing both groups from dropping objects. This provides centralized governance with Azure RBAC for storage access and Unity Catalog for table-level authorization."

Core Databricks: "Secret Scopes and Spark configurations only provide authentication to external resources such as ADLS. They do not automatically grant Databricks permissions. Workspace access, cluster permissions, and SQL Warehouse permissions must be granted separately within Databricks. Typically, I assign Azure RBAC roles such as Storage Blob Data Reader or Contributor for storage access, and Databricks permissions such as Workspace User, Cluster Attach, and SQL Warehouse Use depending on the user's responsibilities." Workspace Access Grant: Workspace User-> Allows: Login Create notebooks Run notebooks. 
Cluster Access: Go to Cluster ->Compute ↓ Cluster ↓ Permissions ->Grant: CAN ATTACH TO or CAN RESTART or CAN MANAGE ->Example: Developer Group ↓ CAN ATTACH TO. 
SQL Warehouse Access Go to: SQL Warehouses ↓ Warehouse ↓ Permissions ->Grant: CAN USE Example: HR Group , Developer Group.
